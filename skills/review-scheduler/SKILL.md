# Review Scheduler（復習提案スキル）

SM-2間隔反復アルゴリズムにより、学習した内容を「忘れかける直前」に復習提案する。
データはNotion DBに永続化し、セッションをまたいで復習スケジュールを管理する。

## 前提知識

SM-2アルゴリズムの詳細仕様は `references/sm2-algorithm.md` を参照すること。
ここではスキルの使い方とワークフローに集中する。

## sm2_compact フォーマット仕様

SM-2の4つの状態値を `|` 区切り1行に圧縮する。
これにより、50項目あっても個別プロパティ方式と比較して約200プロパティ分のトークンを削減できる。

**形式**: `n{repetitions}|ef{ease_factor}|i{interval}|q{last_quality}`

| プレフィックス | SM-2変数 | 型 | 初期値 |
|--------------|----------|-----|--------|
| n | repetitions（連続正解回数） | int | 0 |
| ef | ease_factor（覚えやすさ係数） | float（小数第2位） | 2.5 |
| i | interval（現在の間隔・日数） | int | 0 |
| q | last_quality（最新の評価） | int (1-5) or `-` | - |

**例**:
```
初期状態:         n0|ef2.5|i0|q-
1回目復習後(q=4): n1|ef2.5|i1|q4
2回目復習後(q=5): n2|ef2.6|i6|q5
3回目復習後(q=4): n3|ef2.6|i16|q4
不正解でリセット: n0|ef2.18|i1|q2
```

**パース**: `sm2_compact` を `|` で分割 → 各要素の先頭プレフィックスを除去して数値化
**エンコード**: SM-2計算後の新しい値を同じ形式で連結

## モード1: 学習項目の抽出・登録

**トリガー**: ユーザーが「これ覚えておいて」「復習登録して」と言ったとき、
または技術的な学習を含む会話の終盤でスキルから提案するとき。

### Step 1: 会話から学習項目を特定する

会話の中から以下に該当する内容を抽出する：

- **新しく学んだ概念**: 初めて知った用語、仕組み、原理
- **理解が曖昧だったもの**: 質問して初めてわかった内容
- **よく間違えるパターン**: 勘違いしていたことの訂正
- **重要な区別**: AとBの違い、使い分けの基準

抽出の粒度は「1つの質問で復習できるサイズ」を目安にする。
大きなトピック（例: "SAP SD全体"）ではなく、具体的な知識単位（例: "受注伝票タイプORとTAの違い"）に分割する。

### Step 2: ユーザーに確認する

抽出した項目をリストで提示し、登録するかどうか確認を取る：

```
📚 以下の学習項目を復習リストに追加しますか？

1. 【SAP】受注伝票タイプORとTAの違い
   → ORは標準受注、TAは電話受注。在庫確認の挙動が異なる。

2. 【Laravel】whenメソッドとif-elseの使い分け
   → whenはクエリビルダ内で条件分岐。nullチェックにはif-elseが明示的。

追加する項目を選んでください（全部 / 番号指定 / なし）
```

### Step 3: Notion DBに登録する

確認が取れた項目を `notion-create-pages` でDBに追加する。

各項目の初期値：
- `sm2_compact`: `n0|ef2.5|i0|q-`
- `sm2_next_due`: 今日の日付
- `status`: active
- `review_history`: `[]`
- `source_chat_date`: 今日の日付

登録完了後、次の復習タイミングを伝える：
「明日から復習対象になります。会話の最初に挨拶してもらえれば自動チェックします。」

## モード2: 復習チェック・提案

**トリガー**: 会話開始時の挨拶、または「復習ある？」と聞かれたとき。

### Step 1: 期日到来の項目を取得する

Notion DBから `status == 'active'` かつ `sm2_next_due <= 今日` の項目を検索する。

```
notion-search で「📚 復習スケジューラ v2」DBのdata_source_urlを使って検索
data_source_url: collection://697adc47-ddf3-4dee-9b35-61d70719726f
```

### Step 2: 復習を提案する

期日到来の項目がある場合：

```
📝 今日の復習が {N}件 あります！

1. 【SAP】受注伝票タイプORとTAの違い（n2|ef2.5|i6|q4）
   ヒント: 在庫確認の挙動に注目

2. 【Laravel】whenメソッドとif-elseの使い分け（n1|ef2.5|i6|q3）
   ヒント: クエリビルダの文脈で考える

覚えてる内容を自分の言葉で説明してみてください。
```

期日到来の項目がない場合：
```
✅ 今日の復習はありません！次の復習は {最も近い項目} が {N日後} です。
```

### Step 3: ユーザーの回答を評価する

ユーザーが復習回答をしたら、以下の基準でqualityを判定する：

| ユーザーの応答パターン | quality | 対応 |
|----------------------|---------|------|
| 正確に説明できた、補足もできた | 5 | 「完璧です！」 |
| 概要は合っているが詳細に曖昧さがある | 4 | 足りない部分を補足 |
| かなり苦労したが最終的に思い出した | 3 | 要点を再整理して提示 |
| 「なんだっけ？」→ ヒントで思い出した | 2 | 全体を復習として再説明 |
| 全く思い出せない | 1 | 最初から丁寧に再説明 |

**重要**: qualityの判定はユーザーに押し付けず、応答の内容からClaudeが判断する。
ただし「これは4（良好）で記録しますが、違う評価にしたい場合は教えてください」と
ユーザーが修正できる余地を残す。

### Step 4: SM-2で次回スケジュールを計算する

1. `sm2_compact` をパースして現在の状態を取得
2. `references/sm2-algorithm.md` のアルゴリズムに従って新しい状態を計算
3. 新しい `sm2_compact` 文字列をエンコード

**計算例**:
```
現在: n1|ef2.5|i1|q4
今回: quality = 5

→ q >= 3 かつ n=1 なので interval = 6
→ EF = 2.5 + 0.1 = 2.6
→ repetitions = 2

結果: n2|ef2.6|i6|q5
next_due = today + 6日
```

### Step 5: Notion DBを更新する

計算結果で該当項目を更新する：
- `sm2_compact`: 新しい圧縮文字列
- `sm2_next_due`: 計算した次回復習日
- `status`: quality >= 4 が5回連続 → `mastered` に変更
- `review_history` に今回の記録を追記: `[..., {d:"0318",q:4,i:6}]`
  （review_historyも日付を短縮形で記録してトークンを節約）

更新後、結果を報告する：
```
📊 復習結果:
  評価: 5（完璧）
  SM-2: n1|ef2.5|i1|q4 → n2|ef2.6|i6|q5
  次回復習: 2026-03-24（6日後）
```

## モード3: 学習状況の確認

**トリガー**: 「復習リスト見せて」「学習状況は？」と聞かれたとき。

Notion DBから全itemを取得し、`sm2_compact` をパースしてサマリーを提示する：

```
📊 学習状況サマリー

【アクティブ】12件
  今日の復習: 3件 | 今週の復習予定: 7件
  
【カテゴリ別】
  SAP: 5件 | Laravel: 4件 | ABAP: 2件 | AI: 1件

【習得済み】4件（5回連続正解）

【最も忘れやすい項目】（ef値が低い順）
  1. BAPI_SALESORDER_CREATEのパラメータ構造 (n2|ef1.5|i3|q3)
  2. Eloquent withとloadの違い (n1|ef1.8|i1|q2)
```

## モード4: 項目の管理

**トリガー**: 「この項目を停止して」「復習から外して」「再開して」等。

- **停止**: `status` を `suspended` に変更（復習対象から外れるが削除はしない）
- **再開**: `status` を `active` に戻し、`sm2_next_due` を今日に設定
- **削除**: ユーザーの明示的な指示があった場合のみ、Notionからアーカイブ

## 自動抽出のガイドライン

会話中に以下のパターンを検知したら、会話の終盤で復習登録を**提案**する（勝手に登録はしない）：

1. **「つまり〇〇ということ？」の確認パターン**: ユーザーが理解を言語化した場合
2. **「知らなかった」「初めて知った」**: 新規知識の獲得
3. **コード例の解説後**: 原理を伴う技術説明があった場合
4. **間違いの訂正**: 誤解を正した場合（最も復習効果が高い）

提案は会話の邪魔にならないよう、自然な区切りで行う。
会話の途中で突然「復習登録しますか？」と割り込まないこと。

## 注意事項

- 復習はあくまで**提案**であり、ユーザーに強制しない
- 挨拶時の自動チェックは簡潔に（1〜2行で済ませる。詳細は聞かれたら）
- 項目が多すぎる場合は優先度の高いもの（期日が過ぎている、EFが低い）を上位5件に絞る
- Notion APIの呼び出しは最小限にまとめる
- SM-2の計算はClaude側で行い、結果だけNotionに書き込む
- 旧DB（📚 復習スケジューラ v1: 846bbeb4fe634e1da05cf9a4aa6246ba）は使用しない
- **notion-searchの制約**: セマンティック検索のため、登録直後の項目がインデックスに
  反映されていない場合がある。復習チェック時にsearchで期待した件数が返らない場合は、
  `notion-fetch` で data_source_id（`collection://697adc47-ddf3-4dee-9b35-61d70719726f`）
  を直接fetchしてDB全体のスキーマを取得し、個別ページIDからfetchする方法にフォールバックする
