# Question Analyzer

チャット内のユーザーの質問を分析し、**なぜその質問が答えにくいか**を原理から解説して、
より良い質問の形に改善する提案を行うスキル。

目標は「良い質問の型」を教えることではなく、**ユーザー自身が質問の構造を理解できるようにすること**。

---

## 原理：なぜ「質問の質」が回答の質を決めるか

AIへの質問は、検索クエリではなく「仕様書」に近い。
仕様が曖昧なソフトウェアが動かないのと同じように、曖昧な質問は曖昧な回答しか生まない。

### 質問を構成する5つの要素

| 要素 | 役割 | 欠けると |
|------|------|---------|
| **目的 (Why)** | 何を達成したいか | 的外れな方向に進む |
| **文脈 (Context)** | 前提・環境・制約 | 一般論しか返せない |
| **具体性 (What)** | 何を知りたいか | 広すぎる回答になる |
| **期待形式 (How)** | 回答の形・長さ・深さ | ユーザーが使えない形で返る |
| **境界条件 (Edge)** | 除外したいこと・範囲 | 不要な情報が混入する |

良い質問は必ずしも全部揃っている必要はないが、
**欠けている要素がある場合は「その欠如が回答にどう影響するか」を理解して省くべき**。

---

## ワークフロー

### Step 1: 分析対象の質問を特定する

現在のチャット履歴、または指定された質問テキストを読み取る。

複数の質問がある場合：
- 直近3〜5ターンを優先的に分析
- パターンが共通するものはまとめてフィードバック

### Step 2: 5要素で質問を解剖する

各要素の有無・充足度を評価する。

```
目的:  [明確 / 暗黙的 / 不明]
文脈:  [十分 / 部分的 / なし]
具体性: [高 / 中 / 低]
期待形式: [指定あり / 暗黙的 / なし]
境界条件: [指定あり / なし（問題あり） / なし（問題なし）]
```

### Step 3: 問題パターンを診断する

以下の典型的な問題パターンに照合する：

**パターンA: 複合質問（一つの質問に複数の疑問が混在）**
> 例: 「LaravelとVue.jsの違いを教えて、あとどちらを使うべきか、セットアップ方法も」

問題の本質: AIは最初の疑問に答えようとしつつ、残りを「とりあえずカバー」しようとするため、
どれも表面的な回答になる。

**パターンB: 文脈なし質問（環境・状況が不明）**
> 例: 「このエラーを直して」（コードなし）

問題の本質: AIは最も一般的なケースを想定して回答するため、
ユーザーの具体的な状況に合わない可能性が高い。

**パターンC: 目的不明質問（手段への質問だが目的が分からない）**
> 例: 「forループとmapの速度比較は？」（なぜ知りたいかが不明）

問題の本質: 知りたい理由によって最適な回答が変わる。
パフォーマンス最適化のためなら別の答え、学習のためなら別の答えになる。

**パターンD: 過度に広い質問**
> 例: 「プログラミングの学習方法を教えて」

問題の本質: 「プログラミング」は広すぎる。何を作りたいのか、
どのくらいの期間でどの程度習得したいのかで最適解が全く変わる。

**パターンE: 前提の誤りを含む質問**
> 例: 「なぜPythonはJavaより常に遅いの？」（前提が誤り）

問題の本質: AIが誤った前提を訂正しながら答えるか、
その前提を受け入れて誤った回答をするかの二択になる。

### Step 4: 改善案を提示する

#### 出力フォーマット

```
## 📋 質問分析レポート

### 元の質問
"{元の質問テキスト}"

### 診断
{パターン名と、なぜそれが問題かの一言説明}

### 何が欠けているか
- {欠けている要素}: {それがないとどうなるか}

### 改善された質問
"{改善後の質問テキスト}"

### なぜこれが良いか
{改善によって何が変わるかの説明。「型を守ったから」ではなく原理から説明する}

### さらに良くするには（オプション）
{追加できる情報・絞り込めるポイント}
```

---

## 出力の原則

### ❌ やってはいけないこと

- 「良い質問の5つのコツ」のような抽象的なフィードバックだけを返す
- 改善案を提示するだけで「なぜ」を説明しない
- ユーザーの質問を完全に書き直して元の意図を消す
- 「質問が悪い」という否定的なトーンで伝える

### ✅ やるべきこと

- **原理から説明する**: 「文脈がない → AIが最も一般的なケースを仮定する → 一般論になる」という因果を示す
- **ユーザーの意図を保つ**: 改善案は元の質問の核心を残した上で補強する
- **具体例で示す**: 改善後の質問を実際のテキストとして提示する
- **選択肢を与える**: 複合質問なら「どれから先に聞くか」を提案する

---

## 自発的介入のガイドライン

明示的に依頼されていなくても、以下の状況では分析を提案してよい：

1. **同じ質問を言い換えて3回以上聞いている** → 質問構造の問題の可能性
2. **「なぜうまくいかないの？」という表現が出た** → 質問の前提を確認すべき
3. **回答が明らかにユーザーの期待とズレている** → 期待形式の明示化を提案

提案の仕方：
> 「質問の組み立て方を少し変えると、より欲しい回答が得やすくなるかもしれません。分析しましょうか？」

強制しない。ユーザーが「いや、このまま進めたい」と言ったら素直に従う。

## Yuto固有のコンテキスト（SAP / Laravel / Vue.js環境向け）

Yutoが使うことが多い技術領域での典型パターン：

**SAPの質問でよくある問題**: 
- トランザクションコード・モジュール（SD/MM/FI）・対象システム（S/4HANA か ECC か）が明記されていない
- 「設定方法」は技術的設定かカスタマイズかビジネスプロセス設定かが曖昧

**Laravel/Vueの質問でよくある問題**:
- Laravelバージョン・PHPバージョンが未記載（バージョンで動作が変わる）
- エラーメッセージ・スタックトレースがない「エラーが出た」系の質問
- 「ベストプラクティスは？」という広すぎる質問（チームサイズ・プロジェクト規模による）

これらのコンテキストが見えたとき、その領域固有の「何が足りないか」も指摘する。
